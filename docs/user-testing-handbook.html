Complete beginner frontend testing handbook

This is a practical training path for someone who has never used the current HABIS frontend. Follow it from top to bottom on a test database, then repeat individual scenarios using the checklists. Every field is explained in plain language, with an example and the reason the application needs it.

Important: This guide is for testing. Use fake names, fake phone numbers, fake documents, and test money. Never paste production passwords, OTPs, identity documents, or customer data into a test record.

Frontend: run the Next.js project in ~/personal_projects/habbis-front-local and open its local URL (normally http://localhost:3000). API: the frontend talks to the configured API proxy; if a page says “network error,” check the frontend .env and the API health first. The sidebar is permission-aware: a missing menu item usually means the signed-in user does not have the required permission.

How the application fits together

HABIS is a financial operations system. Most workflows are deliberately staged. A record may be created as a draft, reviewed, approved, posted, reconciled, or closed. The application often prevents a later step until the earlier step is complete. That is a control, not a bug.

ConceptBeginner meaningWhy it matters while testing
AgencyA branch or operating unit.Most users and financial records are scoped to one agency. A user in Agency A should not see Agency B records.
Accounting dayThe business-date container for registrations and close.Cash sessions and journals inherit a business date. Keep the target day open while creating or changing financial records; a missing, closed, or closing day blocks most mutating requests.
TillA physical or logical cash drawer.A till must be active and daily-closed before opening a new teller session.
Teller sessionOne teller's use of one till for one business date.Open it, post operations, reconcile cash, then close it.
Customer accountA ledger-backed account belonging to a client.Deposits, withdrawals, recoveries, and transfers need a valid active account and correct ledger mapping.
Journal entryAn accounting event with balanced debit and credit lines.It may move through draft, submitted, approved, posted, rejected, or reversed states.
Public IDThe long identifier shown by the API, not the internal database integer.Support and API examples use public IDs. Copy the entire value when filtering or opening a detail page.
Golden rule: When a form asks for a date, amount, agency, account, or person, choose the record that belongs to the same agency and accounting day as the workflow you are testing.
Important order rule: Before creating or changing reference data, open an accounting day for the required scope. In the current API, denomination, till, sector, sub-sector, operation-code, mapping, and other mutating setup requests are protected by the accounting-day registration lock. A closed, closing, or missing day will block the save even when the user has the feature permission.
Dependency order for a clean first run: create the agency and staff → open the accounting day → create denominations and ledger accounts → create operation codes and account mappings → create account products → create and verify the client → open the customer account → create the loan product → create, approve, charge, and disburse the loan.
Scope exception: denominations are global (not agency-owned). A platform administrator must open an institution-scoped accounting day before creating them. Agency-scoped days are used for agency-owned financial records.

Prepare a safe test environment

Recommended test data

  • One platform administrator for setup and permission tests.
  • One agency manager, one teller (the current configuration uses teller; it does not define a separate cashier role), one accountant, one loan officer, and one KYC officer.
  • Two agencies so cross-agency access can be tested.
  • At least two denominations (for example XAF 1,000 and XAF 500). Include a zero-count line when reconciling a zero-balance till.
  • One active cash ledger account per till and the ledger accounts required by deposit, withdrawal, loan, and journal operations.
  • One client with a verified identity document, one guarantor, one proxy, and one active customer account.
  • A loan product with a small amount and short term, such as 100,000 XAF over one month, for quick testing.

Keep a test log

For each scenario write down: date/time, signed-in user, agency, page, data used, expected result, actual result, screenshot or public ID, and whether the result was expected. This makes a failure reproducible instead of becoming “it did not work.”

Record IDs as you go: agency public ID, user public ID, till public ID, session public ID, client public ID, account public ID, loan public ID, journal public ID, and batch-run public ID.

Language and amounts

The interface supports English and French. Switch language before testing validation messages. Monetary APIs use minor units; the frontend money field normally displays major units and converts internally. Always verify the displayed currency and the resulting balance. For XAF in this application, confirm the displayed amount rather than guessing whether a field expects cents.

Coverage boundary

Do not mistake this handbook for an API inventory. It covers the workflows currently reachable from habbis-front-local: dashboard, administration, cash, CRM, credit, accounting, reports, notifications, and backups. The API also contains modules for which this frontend currently has no page.
AreaCurrent handbook coverageWhat is not covered here
Frontend pagesAll currently navigable pages, including their main create, update, review, close, and reporting paths.Backend actions that have no corresponding frontend screen.
API-only modulesNot covered by beginner UI steps.Currency exchange, HR/payroll, insurance, Islamic finance, and regulatory reporting routes. These require an API client and a separate contract/workflow test plan.

For API-only work, start from the matching route file in routes/api/v1/, inspect the request/response types, seed only synthetic records, and preserve the same accounting-day, agency-scope, approval, audit, and idempotency rules described in this handbook.

Users, roles, and permissions

A role is a bundle of permissions. A permission answers “may this user perform this action?” A role does not automatically override agency scope. Create separate test users and never use the platform administrator to test ordinary staff behavior.

platform-adminInstitution-wide setup and emergency administration.Can select agencies, manage permissions, inspect audit and database operations. Verify that it can see multiple agencies but must still choose the intended target for writes.
user-adminStaff-user and role assignment administration.Create/activate staff without granting protected platform-admin access. Test forbidden role escalation.
agency-managerAgency operations and staff/till oversight.Can work inside the assigned agency; cannot read or mutate another agency.
regional-managerRead-only institution or regional oversight.Compare agency filters and confirm that visibility does not imply write permission.
teller / cashierDaily cash operations.Open their permitted session, post deposits/withdrawals, reconcile, and close their own session.
accountantAccounting days, journals, reports, and controls.Start and close days, create/review/post journals, and verify close blockers.
loan-officerClient loans and servicing.Create loans, generate schedules, prepare the approval path, and follow setup-charge/readiness rules.
kyc-officerIdentity and KYC review.Submit, review, verify, reject, suspend, and archive KYC records according to permissions.
compliance-officerCompliance and control review.Review KYC and the compliance step of loan approvals; do not assume this role can create loans.
auditorRead-only audit and oversight.Inspect evidence and reports, then confirm that mutation controls return 403.
staffBaseline authenticated staff.Verify the minimum visible menu and negative authorization cases.

Permission test pattern

  1. Sign in as the role under test.
  2. Confirm the expected menu item appears.
  3. Try the permitted action in the correct agency.
  4. Try the same action in another agency and expect an empty result, forbidden response, or validation error.
  5. Sign in as a role without the permission and confirm the page/action is hidden or returns HTTP 403.

1. Sign in and activate accounts

Using the test-staff seeder? The accounts created by TestStaffAccountsSeeder are already active, phone-verified, and password-ready. Skip the activation wizard for those identities and use the credentials printed by the seeder. Use the activation steps below for a newly invited/pending user.

First activation

  1. Open Activate account from the login page.
  2. Enter the phone number exactly as registered. Spaces are normalized; use the country code consistently, for example +237650000001.
  3. Request the activation code. In a test environment configured with a log provider, find the OTP in the configured application log or test mail/SMS sink.
  4. Enter the OTP before it expires and before the maximum failed attempts are consumed.
  5. Choose a password and confirm it. This step changes the account from pending_verification to active, records phone verification, and records activation time.
  6. Return to login and sign in with the phone number and new password.

Login fields

FieldWhat to enterCommon mistake
PhoneThe registered staff phone number.Using a client phone, a number with a missing country code, or a number belonging to a suspended user.
PasswordThe password created during activation or password reset.Repeated wrong attempts trigger rate limiting. Wait for the countdown instead of repeatedly submitting.

Forgot password

Choose Forgot password, enter the registered phone, retrieve the password-reset OTP, enter it, and choose a new password. A user must be active and phone-verified for this flow. If the account is suspended or deactivated, an administrator must resolve its status first.

Testing shortcut warning: manually marking an account active without setting a password does not make it immediately usable. The user must complete password reset before login.

3. Reference data

Prerequisite — do this first: Go to /admin/accounting-day and open the intended scope. For denominations, choose institution in the platform-admin scope selector; an agency-scoped day alone will not satisfy the global denomination request. For agency-owned setup, choose the matching agency. Return here only after the day shows open (or reopened). If you see an error such as accounting_day_missing, accounting_day_closed, or accounting_day_closing, stop and resolve the day status before retrying the form.

4. Cash operations: the till workflow

Never skip the order: accounting day open → till/session open → transactions → no pending transactions → reconciliation with zero difference → session close → accounting-day close. Closing the session before reconciliation creates a day-close blocker.

Accounting day — /admin/accounting-day

For the first run, open scope = agency, select Test Agency A, and enter the business date shown on the screen (for example 2026-07-16). Leave the date empty only when you intentionally want the API to choose the next date. Use scope = institution only for global records such as denominations.

  1. Choose the agency scope (or institution scope if your role permits it).
  2. Open a day with the intended business date. The business date is not necessarily today's calendar date.
  3. Confirm the status is open and registration is allowed.
  4. During close, use Start close first. This runs preflight checks and moves the day to closing.
  5. Resolve every blocker shown: open sessions, pending cash transactions, unposted journals, missing reconciliations, or failed close-control batches.
  6. Choose Close only after the preflight is clean.
  7. Use Cancel close when a day is in closing but work must continue. Use Reopen only with the proper permission and an explicit reason.

First deposit example: choose the customer's active XAF savings account, amount 10,000 XAF, currency XAF, operation code cash_deposit, payment method cash, cash amount 10,000 XAF, denomination count 10 × XAF-1000, initiator holder, description Test deposit, and idempotency key deposit-test-001. Submit once, then inspect the posted transaction and journal. Reusing the same key should return the existing operation; changing the amount with that key should fail.

Before closing, every transaction must be posted, reversed, or cancelled. A status such as pending or pending_review blocks close.

Reconciliation, then close

  1. Open the session's reconciliation action.
  2. Count the physical cash by denomination. The total must equal the session's current theoretical balance.
  3. Enter a note explaining unusual circumstances, not a secret.
  4. Submit and confirm balanced and difference 0.
  5. Open the session close action. For a denomination-required till, enter the same physical count; otherwise enter the closing amount.
  6. The closing declaration must equal the posted theoretical balance. The API now also requires a current balanced reconciliation, so the sequence is enforced server-side.
Example: opening cash 50,000 + deposits 20,000 − withdrawals 5,000 = theoretical 65,000. Count notes that total exactly 65,000, record reconciliation, then close with 65,000.
Zero-balance example: reconciliation still requires at least one denomination line in the frontend/API. Select an active denomination and enter count 0. The actual total is zero; then close with amount zero.

Cash inspection — /operations/inspection

Use inspection to filter sessions by date, till, teller, and status, open a session detail, compare opening/closing declarations and timestamps, and review its reconciliation history. For an open session, the inspection page can create a reconciliation when your role has cash.reconciliations.manage; use it to reproduce the same balanced/unbalanced cases before attempting close.

Guarantors, proxies, and guarantees

A guarantor is a person or organization supporting a client. A proxy is authorized to act for a client. A guarantee obligation connects a guarantor/proxy to a loan or account and has its own amount, status, and release lifecycle. Test expiry, revocation, wrong-client links, and unauthorized actions.

The standalone directories at /guarantors and /proxies are read/filter views; create and verify records from the client detail tabs. Confirm that opening a directory does not grant permission to edit another agency's record.

Schedule and setup charges

  1. Generate the schedule after the loan reaches the state that permits it.
  2. Inspect principal, interest, due dates, totals, and final rounding. For LP-TEST-01, expect principal 50,000 XAF, one installment, and zero interest/fees.
  3. Open setup-charge readiness. With the starter product, there should be no required charge. If you create a fee-bearing product later, assess the charge before disbursement.
  4. Collect a charge from an eligible customer account or open teller cash session only when the product requires it; follow the approved exception/waiver flow if permitted.

Disbursement — /credit/disbursement

For the first test choose the cash channel, select the open session/till, and disburse 50,000 XAF only after the till has at least that much posted cash. The till must be active, open, agency-matching, ledger-mapped, and have sufficient balance. For a transfer test, first link an active customer account belonging to the same client, agency, and currency. Confirm the loan status, journal, teller transaction (cash), account balance (transfer), notification, and audit event.

Loan transfers — /credit/transfers

Select one eligible loan, choose an active manager in the same agency, enter a reason such as Portfolio balancing, and use the open business date. Preview before submitting. Test that disbursed/closed-ineligible statuses, inactive or wrong-agency targets, blank reasons, and duplicate submissions are rejected.

Delinquencies and recovery — /credit/delinquencies

Select a client first, then a loan that is overdue or otherwise recoverable. For a tracking use a follow-up date four days after the open business date, reason Payment reminder, an appointment one day later, promised amount 5,000 XAF, currency XAF, and a factual comment. Then add its item/evidence record. Test real-estate and personal-guarantee types separately; release only after the loan is closed and the permission allows it.

Repayments and write-offs

The current frontend has no standalone repayment or write-off page. Treat these as API-only workflows until a frontend route is added; do not claim that the loan pages alone exercise them. Use the corresponding backend route tests with synthetic data and preserve accounting-day, balance, allocation, and authorization checks.

7. Accounting and journals

Ledger accounts — /accounting/ledger-accounts

A ledger account is a shared accounting bucket, not a customer record. You do not create one account for every customer, loan, or denomination. Create a small chart for the agency you are testing, then link the relevant account to a till, account product, loan product, or operation mapping.

Before you click Create: keep an accounting day open for the target agency. In the current safe slice every new ledger account must be attached to an agency. A platform administrator must explicitly select the target agency; leaving the agency blank will be rejected. Create the parent accounts first, but leave Parent blank for the starter chart unless you already have a deliberate hierarchy.

Exactly how to create the starter chart

  1. Sign in as platform-admin or an authorized accounting user and open the accounting day for the agency you will test.
  2. Open /accounting/ledger-accounts and choose Create. For each row above enter the code, name, class, and normal side; choose the same agency every time; set Status to active; leave Account type and Parent blank for the first pass.
  3. Open the account-product page and create a savings/current product linked to 2100. Create a loan product linked to 1200. Do not link a product to an inactive account or to an account from another agency.
  4. Create the till and assign 1000 as its cash ledger account. If you have two tills, create two cash accounts (for example 1000-A and 1000-B) so their balances cannot be confused.
  5. For a simple cash deposit/withdrawal, the till account plus the customer-account ledger are sufficient. For cheque or transfer tenders, create active, approved mappings for cash_deposit_cheque, cash_withdrawal_cheque, cash_deposit_transfer, or cash_withdrawal_transfer as applicable.
  6. For loans, open the mappings tab and use the readiness result. A typical loan setup needs an approved active debit mapping for loan_principal_disbursement pointing to 1200, plus credit mappings for each repayment component you enabled: loan_repayment_principal, loan_repayment_interest, loan_repayment_fees, loan_repayment_tax, and loan_repayment_penalty. Link each to the matching asset, revenue, or liability account and use the same agency, currency, and effective date as the loan.
  7. Run one small cash deposit first, then inspect the journal: the till is debited and the customer-deposit ledger is credited. Only after that works should you add fees, taxes, non-cash tenders, or loan repayment mappings.

A procedure is a registered job definition; a run is one execution for a business date and agency. For a safe first test, create or reuse cash_close_verification with name Cash close test, description Verify test sessions before close, schedule manual, and empty metadata. 8. Reports, audit, notifications, and backups

Dashboard and reports

Use the dashboard first to establish a baseline. Then compare the relevant report with the underlying detail: /reports/par (portfolio at risk), /reports/journal, /reports/cash-draft, /reports/exigible, and /reports/balance. Filter by agency, date, currency, and status. A report total should be explainable from posted records, not pending drafts. /reports/release is visible in the source navigation but intentionally unavailable in the current frontend; do not use it as a test step.

Notifications — /notifications

Open the notification bell and notification page after actions such as session open/close, reconciliation accepted/rejected, loan decisions, and batch failures. Mark one notification read and then mark all read. Verify unread count changes and confirm that marking read does not alter the underlying financial record.

Database backups — /database/backups

Test only with an authorized administrator. Create a backup, wait for its status, verify it, inspect storage metadata, and test the restore plan in dry-run or verify-only mode before any replace/restore action. A restore can destroy current data; never test replace against the only copy of a database.

Do not use production credentials in this handbook, screenshots, browser history, issue tickets, or examples. Redact phone numbers, emails, tokens, OTPs, and customer identity documents.

How to report a bug

  1. Write the exact URL/page and language.
  2. Record the user role, agency, business date, and public IDs.

  4. Capture the smallest reproducible sequence.
  5. State expected versus actual result.
  6. Attach audit event IDs, transaction/journal/session IDs, and a redacted screenshot.

Field glossary

Active, inactive, suspended, deactivated, archived

Active can normally participate in the workflow. Inactive is configured but unavailable for normal use. Suspended is deliberately blocked pending review. Deactivated is disabled as a user/account. Archived is retained for history but should not be used for new work.

Business date versus calendar date

The business date is the date under which accounting/reporting occurs. It can differ from the server's current date during backdated testing or a reopened day. Always use the accounting day shown by the application.

Opening/closing declaration

The amount the teller declares is physically present at the start/end of a session. It is not automatically the customer's account balance and is not a journal line by itself.

Theoretical balance

The system-derived till balance: opening cash plus posted cash deposits minus posted cash withdrawals. Pending transactions are not included.

Actual/countable balance

The total obtained from physical denomination counts. It must equal the theoretical balance for a balanced reconciliation.

Difference

actual balance − theoretical balance. Zero is balanced. A positive value is surplus; a negative value is shortage. The current close control requires zero.

Ledger account versus customer account

A customer account is the product-facing account a client owns. A ledger account is the accounting book account used to post debits and credits. A customer account normally links to a ledger account.

Debit and credit

Accounting sides, not synonyms for “payment” and “receipt.” Whether a debit increases or decreases a balance depends on the account's normal balance side.

Idempotency key

A unique request key used to prevent the same operation from being posted twice when a user retries after a timeout. Reusing it for a different operation should fail.

Maker/checker

Separation of duties: one person prepares or submits an action and another authorized person reviews or approves it. Test that the workflow does not silently let one person skip the required review.

Public ID

A non-sequential identifier exposed to the frontend/API. Use it in URLs and support notes; do not rely on internal database IDs.
